# План рефакторингу плагіну RCListener

## Поточний стан
- Весь функціонал (сканування портів, розбір протоколу, передача UDP, UI-індикація, кешування налаштувань) зосереджений у єдиному класі `RCListener`, що ускладнює супровід і тестування.
- Багатопоточність реалізована вручну через `Thread`, блокування `lock` та `ManualResetEventSlim`; відсутні явні життєві цикли задач і механізм координації зупинки.
- Логіка сканування портів та WMI-фільтр змішані з логікою handshake і обробкою таймаутів, що робить контроль станів складним.
- Парсер серійних даних формує буфер, шукає «$RM,» і одночасно змінює стан каналів, EMA-фільтр і виклик дій для gimbal — важко відокремити помилки протоколу від бізнес-логіки.
- Обробка діапазонів каналів (19–24) та мапінг на дії гімбала рознесені по коду без окремих шарів конфігурації та виконання команд.
- UI-компонент статусу створюється та оновлюється без явної абстракції, із прямими викликами `MainMenu` і роботою з `Bitmap`.

## Цілі рефакторингу
- Розділити відповідальності на ізольовані сервіси з чіткими контрактами.
- Перейти на керовані `Task`/`CancellationToken` замість сирих потоків і ручних блокувань, щоб спростити зупинку та обробку помилок.
- Зробити обробку протоколу та бізнес-логіку (нормалізація каналів, дії для gimbal) незалежними від вводу/виводу.
- Підготувати код до покриття тестами: симуляція серійних кадрів, перевірка мапінгу каналів, CRC та відправки UDP.

## Запропонована декомпозиція
1. **Інтерфейси контрактів**
   - `ISerialPort` / `ISerialPortFactory` для абстракції порту та полегшення тестування.
   - `IRcFrameParser` для виділення парсингу `$RM,...` і перетворення у DTO кадру каналів.
   - `IRcChannelProcessor` для нормалізації, EMA та визначення дій за діапазонами.
   - `IGimbalCommandSender` для побудови пакетів, CRC та відправки UDP.
   - `IPortScanner` для WMI-фільтру та вибору кандидатів.

2. **Сервіси**
   - `PortScanner`: інкапсулює `FilterPortsWithWmi`, кеш останнього порту, логіку перебору кандидатів і таймаути handshake.
   - `SerialSession`: відповідає за відкриття/закриття порту, підписку на події та відстеження здоровʼя (таймаут даних, наявність у `GetPortNames`).
   - `RcFrameParser`: приймає сирі рядки, валідує формат, повертає структуру каналів, генерує події «новий кадр».
   - `ChannelProcessor`: реалізує EMA/клемпінг, мапінг каналів 19–24 на діапазони та видає перелік необхідних команд.
   - `GimbalCommandSender`: зберігає `PacketStore`, додає CRC, надсилає через `UdpClient`, із винесеним налаштуванням IP/порта.
   - `UiStatusPresenter`: створює та оновлює кнопку статусу, реагує на зміни зʼєднання/сканування (події від сервісів).

3. **Координація та потоки**
   - `RcListenerController` як фасад плагіну: композиція сервісів, керування життєвим циклом через `CancellationTokenSource`, запуск фонових задач сканування й відправки RC-кадрів з періодом `SendPeriodMs`.
   - Використання `BlockingCollection`/`Channel` або подій для передачі нових кадрів від парсера до відправника, без спільних мутабельних масивів.

4. **Конфігурація**
   - Винести `RcConfig` у окремий клас/JSON із можливістю підвантаження та валідації (IP, порт, мапінг каналів).
   - Кеш останнього успішного порту та інші налаштування — окремий сервіс `RcListenerSettings`.

5. **Журналювання та телеметрія**
   - Єдиний `ILogger` адаптер (на Console/Debug/Host) з рівнями (Info/Warn/Error), щоб уникнути розпорошених `try/catch` з `Console.WriteLine`.

6. **Тестування**
   - Юніт-тести для `RcFrameParser` (рядки з «$RM,» → масив каналів), `ChannelProcessor` (діапазони → команди), `ComputeCrc16`/`AppendCrc16`.
   - Інтеграційний тест на «вхідний кадр → UDP-команди» через мок `IGimbalCommandSender`.

## Етапи впровадження
1. Створити базові інтерфейси та DTO (кадр RC, події зʼєднання, статус сканування).
2. Перенести WMI-фільтр та кешування порту в `PortScanner`, замінити прямі виклики в основному класі.
3. Виділити `SerialSession` з контрольованим відкриттям/закриттям, адаптувати обробку таймаутів на `CancellationToken`.
4. Розбити парсинг і обробку каналів на `RcFrameParser` + `ChannelProcessor`; RC-потік читає готові DTO, а не сирі рядки.
5. Інкапсулювати UDP-логіку в `GimbalCommandSender`, параметризувати IP/порт з конфігурації.
6. Переписати UI-індикацію через `UiStatusPresenter`, підписати її на події стану (підключено/сканування/помилка).
7. Додати модульні тести на ключові сервісні класи; перевести тимчасові `Thread` на `Task` з токенами скасування.
